# Pre-Sail Index — 문제 정의 · 솔루션 · 계획 · 현황

> 대상 트랙: F4GE Maritime Domain Awareness — **OSINT 기반 해상 회색지대 사전 징후 조기탐지 (Gray-zone Early Warning)**
> 문서 성격: 해커톤 제출·팀 공유용 요약. 모든 수치는 실데이터 실측 결과이며, 미검증·미확보 항목은 그대로 명시함.
> 최종 갱신: 2026-07-04

---

## 1. 문제 정의

아시아·태평양(APAC) 해역에서 해상 민병대의 기습 집결이나 불법 세력의 회색지대(Gray Zone) 도발은 **해상에 진출한 이후에는 저지가 매우 어렵다.** 그러나 이런 행동은 현장에서 우발적으로 일어나지 않으며, 출항 전 단계에서 지상·온라인 전조를 동반한다 — 선원 모집, 배후 항만의 물류·보급 급증, 소셜 미디어 흔적, 그리고 표적 해역으로의 선박 집결.

기존 감시 체계의 한계는 두 가지다.

1. **공통상황도(COP)는 데이터를 화면에 나열할 뿐, 위협을 식별하지 못한다.** 지휘관은 실시간으로 쏟아지는 정보 속에서 무엇이 정상이고 무엇이 전조인지 스스로 판단해야 한다.
2. **신호가 서로 다른 시점에 나타나고, 융합·정량화되지 않는다.** 물리적 선박 집결(위성·AIS)은 대체로 뉴스 보도보다 먼저 발생하지만, 이 선행성을 하나의 지표로 묶어 "지금 이 해역의 출항 임박 위험이 어느 정도인가"를 정량적으로 답하는 도구가 없다.

**핵심 질문**: 출항·집결이 실제로 벌어지기 전에, 공개 정보만으로 "어느 해역이 임박했는가"를 정량적 지수로 조기에 경보할 수 있는가.

---

## 2. 솔루션 — Pre-Sail Index

무료 공개 OSINT를 융합해 관심해역(AOI)별 **출항 임박 지수(0~100)**를 산출하고, 문서화된 실제 집결 사건에 백테스트하여 "지수가 사건에 선행함"을 증명한다.

### 핵심 아이디어
- **신호 융합**: 뉴스 언어 신호(다국어 GDELT) + 물리 신호(GFW AIS 선박 presence, SAR 위성 탐지)를 하나의 지수로 결합.
- **선행성 포착**: 물리 신호가 뉴스에 선행한다는 가설을 지수가 정량화한다.
- **인과성 보장**: 각 시점 `t`의 지수는 오직 `t` 이전 데이터로만 계산한다(180일 트레일링 베이스라인 + 14일 엠바고). 이 규칙이 "지수가 사건에 선행했다"는 백테스트 주장의 신뢰성 근거이며, no-lookahead 단위 테스트로 강제한다.
- **설명 가능성**: 각 지수 값이 어떤 신호에서 비롯됐는지 기여도로 분해해 제시.

### 지수 방법론
1. 신호별로 180일 트레일링 베이스라인(14일 엠바고)에 대한 robust z-score(median/MAD) 계산.
2. 단측 클립 `[0, 6]` — "평소보다 조용함"은 임박 점수에 기여하지 않음.
3. 사용 가능한 신호에 한해 가중 평균(가중치 재정규화) → 포화 변환으로 0~100 지수.
4. 임계값: WATCH ≥ 65, ALERT ≥ 80.
5. 종합 지수 = AOI별 지수의 max (출항 임박은 여러 거점 간 OR 조건).

### 크롤링 없음 · 유료 없음
모든 데이터는 문서화된 공개 API로 취득한다. 웹 스크래핑·유료 피드 미사용. 응답은 로컬 캐시하여 재실행은 오프라인·즉시.

## 3. 초기 계획

## 4. 현재 상태 (정직한 실측 기준 · 통합 후)

### 시스템의 실체
**두 프로젝트(pre-sail index + SEASENTINEL)를 통합한 확장형 데이터 온톨로지 플랫폼이다.** beelink PostGIS를 시스템 오브 레코드로, 실데이터 수집기 다수를 연동하고, 그 위에 트랙1/2 분석과 실시간 대시보드를 얹었다. 합성 데이터는 `simulation/`으로 격리했다.

### 온톨로지 실적재 (실측)
선박 1,561(제재+AIS관측) · 실시간 AIS 포지션(서해) · 항만 3,689(WPI) · 해저케이블 695(TeleGeography) · 사건 19(백테스트 3 + 인용된 실 서해 사건 16) · 문서 1,727(제재+OSINT) · OSINT 215(StealthMole 텔레그램) · 기상 4,842(Open-Meteo) · 신호 7,440 · 지수 1,808(legacy.v0 + index.v1) · 알림 3(다크베슬) · 백테스트 3. 전부 실데이터, provenance 포함.

없는 것(여전히): 상시 경보 스케줄러, 사용자 인증, SAR per-detection 실피드, NLL 지오펜스 공개 좌표.

### 구현 완료
- 실제 API 연동(목킹 아님, 라이브 데이터) · 원자적 파일 캐시
- 인과성 보장 지수 엔진 (no-lookahead 단위 테스트 통과)
- 백테스트 지표(리드타임·피크 백분위·오경보 에피소드) + 차트 6종
- schema 1.0 아티팩트(`data/artifacts/latest.json`) — Phase 2 대시보드의 계약
- 단위 테스트 10개 통과, 워킹트리 클린, 12커밋

### GFW API로 실제 확보한 범위
| 기능 | 결과 |
|---|---|
| 4wings `presence` (AIS 존재) | ✅ **정상**. 선박별·일별 행 반환, 일일 시간 합계로 집계. 세 산호초 모두 사건일 스파이크 확인 |
| 4wings `sar-presence` (SAR) | ⚠️ HTTP 200이나 **거의 빈값**. 산호초 박스가 작고 Sentinel-1 재방문 희소 → 전체 6개 포인트(scarborough만). 실질 기여 없음 |
| Events API (gap·port·encounter·loitering) | ❌ **403 Not authorized**. 토큰 권한 등급에 Events API 없음. 계획 신호 2개(AIS-gap, port-visit) 제거 |

**GFW에서 실질적으로 손에 쥔 신호는 AIS presence 하나.** SAR은 원리상 dark vessel을 잡아야 하나 데이터 부재, Events는 권한 밖.

### 신호 활성 현황 (최종 run 기준)
계획 7신호(Events 2개 제외 후) 중 실데이터가 들어온 것:
- gdelt_place_en, gdelt_militia_en (영어) — 정상
- gdelt_tone — 부분
- gfw_presence_hours — 정상
- gdelt_place_zh, gdelt_militia_zh (중국어) — **0행** (GDELT rate-limit 페널티로 미확보, 재실행 시 채워짐)
- gfw_sar_presence — 6행(무의미)

즉 "다중 소스 융합"의 실체는 현재 **영어 뉴스 + AIS presence 2~3신호**다. 지수 엔진은 사용 가능한 신호에 한해 가중치를 재정규화하므로 결측 신호가 있어도 유효한 지수를 산출한다.

### 백테스트 실결과
| 사건 | 패턴 | lead | peak | 백분위 | 오경보 | 해석 |
|---|---|---|---|---|---|---|
| **Sabina 2024** | 수주 포위전 | **45일** | 95.8 | 100 | 1 | 사건 45일 전 WATCH 발화, 지속 ALERT — 진짜 조기경보 (주력 성공 사례) |
| **Scarborough 2024** | 돌발 물대포 | 0일 | 87.8 | 100 | **0** | 2개월 조용한 기저 → 사건 당일 급상승. 지수 미남발 입증 |
| **Whitsun 2021** | AIS-dark 민병대 | 없음 | 25.9 | 40 | 3 | AIS 차단으로 물리 신호 평탄, 뉴스로만 뒤늦게 발화 — dark vessel 갭 |

**서사**: 장기 집결(sabina)에는 수주 조기경보, 돌발 사건(scarborough)에는 당일 정밀 탐지, dark 민병대(whitsun)에는 AIS 한계가 드러남 — 다중 센서 융합의 필요성을 정직하게 보여줌.

### 계획 대비 편차
1. **GFW Events 403** → AIS-gap·port-visit 신호 제거, 7신호로 재설계.
2. **중국어 GDELT 미확보** → GDELT rate-limit(1회/5초, IP 페널티) 때문. 코드에 스로틀·백오프·빈값 폴백 내장. 쿨다운 후 재실행 시 채워짐.
3. **whitsun 리드 = 없음** (계획 DoD는 리드>0 가정). 실데이터에서 민병대가 AIS를 껐음이 드러남. 리드 시연은 sabina(45일)가 담당. 실패가 아니라 트랙 문제의 본질(dark vessel) 확인.
4. **서브에이전트 미사용** — 이 하네스에서 general-purpose 서브에이전트가 컨텍스트 thrashing으로 반복 중단. 코드는 메인에서 직접 작성하되 태스크별 테스트·DoD·커밋 게이트 유지.

### 프로덕션 급인가 — 아니오
프로덕션에 필요하나 없는 것: 실시간·스트리밍, 서버·API·스케줄러·상시 경보, 지수 방법론 튜닝·통계 검증(현재 3사건 case study, 가중치·임계값 수동), 통합 테스트·CI, 운영 모니터링, 확장 저장소, 견고한 에러 처리.

**가장 큰 격차는 코드 완성도가 아니라 신호 원천의 빈약함이다.** GFW Events 권한과 중국어 뉴스를 확보하지 못하면 "다중 소스 융합" 서사가 실제로는 얇다.

## 5. 다음 단계 (우선순위 순)

1. **신호 원천 확충** — 스텔스몰(텔레그램·다크웹)로 Events 공백을 텍스트 OSINT로 대체, 또는 GFW 상위 권한 토큰으로 Events API 개방. 중국어 뉴스는 GDELT 쿨다운 후 재실행으로 확보.
2. **Phase 2 웹 대시보드** — `latest.json`(schema 1.0) 계약으로 분리되어 있어 별도 세션에서 구현 가능. 붙여넣기·하이라이트·지수 타임라인·기여도 화면.
3. **지수 검증 강화** — 사건 표본 확대, 가중치·임계값 근거 정량화.
